# /project:fix-issue — 修复 Issue

根据 Issue 描述，定位问题代码并生成修复方案。

## 执行步骤

1. **理解** — 阅读 Issue 描述，明确问题范围和验收标准
2. **定位** — 搜索相关代码文件，精确定位问题代码位置
3. **分析** — 分析根因，判断影响范围（是否涉及 API 变更、DB 迁移等）
4. **修复** — 编写修复代码，确保：
   - 修复后不引入新的 lint 错误
   - 现有测试仍然通过
   - 边界条件已覆盖
5. **验证** — 运行相关测试，确保修复有效

## 输出格式

```
## Issue #[编号]: [标题]

### 根因
[一句话描述问题根因]

### 影响范围
- 涉及文件: [列表]
- 是否 Breaking Change: [是/否]

### 修复方案
[修复描述]

### 变更文件
- [文件路径] — [变更说明]

### 验证
- [ ] 单元测试通过
- [ ] Lint 通过
- [ ] 手动验证场景: [描述]
```
